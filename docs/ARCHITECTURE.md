# Архитектура kalve

## Три слоя

Вся система построена на разделении трёх вещей, которые часто смешивают в одну кашу:

```
  ┌─────────────────┐
  │   БРЕНДКИТ      │  Как это выглядит (цвета, шрифты, скругления)
  │   brand.py      │
  └────────┬────────┘
           │
           ▼
  ┌─────────────────┐
  │   ШАБЛОН        │  Что и где на карточке (логика компоновки)
  │   template.py   │
  └────────┬────────┘
           │
           ▼
  ┌─────────────────┐
  │   ДВИЖОК        │  Как именно рисовать (примитивы)
  │   drawing.py    │
  │   typography.py │
  └─────────────────┘
```

**Движок** (низ стека) ничего не знает о бренде или шаблоне. Он умеет рисовать градиенты, скруглённые прямоугольники, пилюли, эмодзи и текст. Функции принимают координаты и цвета напрямую.

**Шаблон** знает структуру конкретного типа карточки: где бейдж, где фраза, где CTA. Он берёт брендкит и контент и дёргает функции движка. Шаблон не содержит hardcoded цветов — всё из брендкита.

**Брендкит** — это чистые данные. Никакой логики. Просто dataclass с цветами, шрифтами и отступами.

## Зачем так

Любые два из трёх можно менять независимо:

- **Новый бренд, старый шаблон.** Пишу `mybrand.py`, запускаю `three_tiers` с ним. Готово.
- **Новый шаблон, старый бренд.** Пишу `google_ads.py` в `templates/`, он использует тот же `SAUNUOLE` брендкит. Вместо каруселей получаю баннеры.
- **Новый бренд, новый шаблон.** Тоже работает, они не связаны.

## Поток данных

```
JSON файл
   │
   ▼
CLI (__main__.py) ─ парсит аргументы, загружает JSON
   │
   ▼
generate() (__init__.py) ─ выбирает шаблон из TEMPLATES
   │
   ▼
template.render(spec, brand, output_dir)
   │
   ├──► typography.font(weight, size) ─ загружает Inter
   ├──► drawing.linear_gradient(...)    ─ рисует фон
   ├──► drawing.rounded_rect(...)       ─ рисует белую карточку
   ├──► drawing.pill(...)               ─ рисует бейдж
   └──► drawing.paste_emoji(...)        ─ вставляет эмодзи
         │
         ▼
      PNG файлы в output/
```

## Почему JSON, а не YAML/TOML

Потому что JSON — это lowest common denominator. Его можно:
- Сгенерировать из N8N Code-ноды
- Написать рукой в Airtable в поле `Cards JSON`
- Отправить через HTTP API
- Валидировать любой схемой

Если хочется YAML — добавить парсер yaml в `__main__.py` на 3 строки.

## Почему Python, а не Node/Rust/Go

- У Pillow (Python) самый зрелый API для text layout и растровой графики
- Хорошо работает с Unicode и сложными скриптами (литовские диакритики, кириллица)
- Быстро писать, легко читать — opensource-инструмент должен быть понятным с первого взгляда
- Аудитория (маркетологи, автоматизаторы в N8N) — скорее Python, чем Rust

## Что НЕ делает kalve (сознательно)

- **Не придумывает контент.** На входе готовый текст, на выходе картинка. Придумывание фраз — отдельная задача (Claude API, ручная работа, банк заготовок).
- **Не публикует.** Генерирует PNG. Публикацией занимается отдельный слой — N8N, ваш скрипт, что хотите.
- **Не хранит историю.** Одноразовая функция: вход → выход. Трекинг «какие карточки уже выпустили» — ответственность того, кто использует kalve.
- **Не делает anti-aliasing нативно.** PIL умеет его за нас, достаточно.

## Что точно появится в будущем

- **Больше шаблонов.** `google_ads` (10 размеров из одного входа), `story` (1080×1920), `og_image` (1200×630 для блогов).
- **Пакетный режим.** Сейчас одна спецификация → одна карусель. Нужен режим «вот 10 спецификаций, выдай 40 PNG».
- **Валидация спецификации.** Сейчас падает с невнятной ошибкой при кривом JSON. Нужна схема и понятные сообщения об ошибках.
- **Поддержка кастомных шрифтов.** Брендкит должен уметь указать свой шрифт, а не использовать только Inter.

## Что вряд ли появится

- **Веб-интерфейс.** kalve — это движок. UI — отдельный проект.
- **Встроенный AI для генерации контента.** У kalve одна задача — рендер. AI-интеграции живут снаружи.
- **Анимации / GIF / видео.** Только статичные PNG/JPG.
